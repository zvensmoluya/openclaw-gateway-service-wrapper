# V2 需求与边界基线

## 背景

当前仓库已经把 OpenClaw gateway 包装成一套相对稳定的单用户 Windows Service 方案，并在实际使用中验证了很多边界条件，包括服务身份、托盘交互、精确停机、remembered config、计划任务桥接，以及围绕当前登录用户权限的约束。

这套方案仍然可用，但也暴露出越来越明显的结构性问题：

- 我们真正想要的是“当前登录用户上下文里的后台运行”，不是“机器级宿主优先”
- OpenClaw 会自行重启，未来也可能调整自己的更新、重启和运行时行为
- 作为包装层，我们不能依赖过多对上游内部行为的猜测
- 当前 Service 模型为了贴近用户语义，已经引入较多补偿逻辑，继续叠加复杂度的收益越来越低

因此，在进入下一阶段重构之前，需要先把真实需求、边界、非目标和关键场景写清楚。本文档作为后续 ADR、V2 架构蓝图和迁移计划的输入基线。

## 文档目标

- 澄清这个项目真正要解决的问题，而不是延续既有实现方式
- 明确 OpenClaw 上游与本仓库包装层各自负责什么
- 列出 V2 必须满足的能力、关键场景与非目标
- 为“从 Windows Service 转向用户级后台宿主”的讨论提供统一语境

## 问题定义

这个项目最初要解决的问题不是“如何做一个 Windows Service”，而是下面这些更贴近实际使用的需求：

- 让 OpenClaw 能在 Windows 上长期后台运行，而不是始终挂着一个前台 PowerShell 窗口
- 让 OpenClaw 尽量使用当前登录用户的权限、配置、目录和网络环境
- 提供稳定的启动、停止、重启、状态与诊断入口
- 支持登录后自启，并保持控制入口足够轻量
- 在不侵入上游实现的前提下，尽量平稳地承接 OpenClaw 的自重启、异常退出和未来更新

## 核心原则

- 用户优先：默认以当前登录用户为运行主体，而不是以机器级服务为运行主体
- 上游优先：不阻断、不重写、不猜测 OpenClaw 的内部生命周期语义
- 包装层克制：包装层只承担宿主、控制、观测和迁移职责，不接管上游业务逻辑
- 生命周期优先：设计先围绕启动、退出、重启、更新、恢复这些场景，再决定 UI 或安装形式
- 低摩擦优先：普通日常使用不应依赖 Service 安装、复杂提权链路或大量系统级补偿逻辑

## 必须满足的需求

### 运行模型

- OpenClaw 必须能在当前登录用户上下文中后台运行
- 默认运行形态不应出现常驻 PowerShell 控制台窗口
- 运行时默认使用当前用户的目录、环境变量、代理和配置路径
- 应继续以单用户 Windows 桌面场景为主要目标，不把多用户并发作为主需求

### 生命周期控制

- 需要稳定的 `start`、`stop`、`restart`、`status` 能力
- 需要明确区分“用户主动停止”和“进程异常退出”这类不同语义
- 需要在宿主重启、托盘退出或控制端重开后继续恢复对当前运行状态的认知
- 需要为 OpenClaw 的自重启、自退出和未来更新预留兼容空间

### 观测与诊断

- 需要持续记录日志，并为用户提供可定位的日志目录
- 需要保留可机器读取的运行状态文件或等价状态接口
- 需要保留轻量诊断入口，便于确认当前配置、运行身份、进程状态和健康检查结果
- 需要让控制面能够在不直接接管进程的情况下读取当前状态

### 自启与控制面

- 需要支持当前用户登录后的自动启动
- 自启应围绕当前用户完成，不应要求把系统级服务作为默认前提
- 控制面可以是托盘、CLI 或轻量设置窗口，但这些控制面不应成为生命周期本体
- 托盘退出不应等同于停止 OpenClaw，除非用户明确发出停止命令

### 迁移与兼容

- 短期内需要允许现有 Windows Service 方案继续作为稳定 fallback
- 新方案需要具备从现有配置、路径和操作习惯平滑迁移的能力
- 迁移过程应尽量避免破坏用户当前可用的 OpenClaw 环境

## 边界与责任

### 上游 OpenClaw 负责

- `openclaw.json` 等上游配置文件的 schema 与语义
- OpenClaw 自身的业务逻辑、运行方式和命令语义
- OpenClaw 未来如何实现自身的重启、更新和版本演进
- OpenClaw 对内部子进程、文件布局和退出码的选择

### Wrapper / Agent 负责

- 提供一个无窗口、当前用户级的后台宿主
- 负责启动、停止、重启和状态查询入口
- 负责日志、状态缓存、健康检查与基础诊断
- 负责登录自启、托盘或 CLI 这类控制面接入
- 负责从旧的 Service 方案向新宿主模型迁移

### Wrapper / Agent 不应假设

- OpenClaw 总会使用固定退出码表达“正常重启”或“更新”
- OpenClaw 的更新总是原地覆盖，或总是采用版本化目录
- OpenClaw 的子进程结构未来保持不变
- OpenClaw 一定会提供稳定的“我要重启”或“我要更新”信号
- 只要 wrapper 能强控进程，就等于掌握了上游全部生命周期语义

## 关键场景

1. 用户登录 Windows 后，后台宿主应能在当前用户上下文中自动启动，并在无可见控制台窗口的前提下拉起 OpenClaw。
2. 用户主动执行 `stop` 时，宿主应把这次退出记录为“主动停止”，避免误判为需要自动恢复。
3. 用户主动执行 `restart` 时，宿主应有明确的停机和重启顺序，而不是简单依赖原地重新拉起。
4. Tray 或其他控制面退出时，后台宿主和 OpenClaw 不应被误停止；控制面重新打开后应能恢复当前状态。
5. OpenClaw 自行退出、重启或替换自身文件时，宿主应尽量以兼容和观测为主，不依赖脆弱的内部假设。
6. OpenClaw 异常崩溃或健康检查持续失败时，宿主应有清晰的状态记录、通知与恢复策略。
7. 用户注销、关机或切换会话时，宿主行为需要明确，至少要保证状态一致、路径明确、重启逻辑可预期。
8. 从当前 Windows Service 方案迁移到新宿主模型时，用户应能保留已有配置和主要操作习惯，且旧方案在过渡期内仍可作为回退路径。

## 非目标

- 不以多用户并发运行作为当前版本主目标
- 不以机器级 Windows Service 作为 V2 的默认宿主
- 不以浏览器控制面或 Web 壳作为必须前提
- 不侵入 OpenClaw 上游内部实现，也不试图接管其版本策略
- 不要求日常使用必须拥有管理员权限
- 不要求 V2 第一阶段就完整复制当前 Service 方案的全部恢复和运维细节

## 设计约束

- 新方案必须优先解决“后台无窗口运行”和“当前用户权限一致性”这两个原始问题
- 新方案应尽量减少对系统级安装、提权、计划任务桥接和服务账户补偿的依赖
- 设计必须把“上游不可控”视为常态，而不是例外
- 控制面、安装器、迁移工具都应围绕后台宿主服务，而不是反过来主导宿主结构

## 成功标准

- 用户不再需要依赖常驻 PowerShell 窗口来运行 OpenClaw
- OpenClaw 默认运行在当前登录用户上下文中，并自然继承该用户环境
- 登录自启、手动启动、停止、重启和状态查询都能稳定工作
- 控制面退出后，后台宿主仍能继续运行并保持状态一致
- OpenClaw 的自重启、异常退出和更新相关场景不再强依赖 Windows Service 语义
- 新方案上线后，旧的 Windows Service 方案可以在一段过渡期内并行存在，并可作为回退选项

## 待确认问题

- OpenClaw 目前以及未来会通过什么外部可观察信号表达“正常重启”或“更新中”
- 用户注销时，OpenClaw 应默认停止、跟随会话结束，还是允许后台宿主参与下一次登录恢复
- V2 的第一阶段是否需要内建自动恢复策略，还是先以状态可观测为主
- 新方案是否需要直接管理 OpenClaw 二进制版本，还是先假定更新由用户或上游独立完成
- 从现有 Service 迁移时，哪些旧配置字段需要继续兼容，哪些字段应明确废弃
