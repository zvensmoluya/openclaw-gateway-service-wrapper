# ADR：V2 默认宿主转向当前用户级后台 Agent

## 状态

Accepted

## 日期

2026-03-24

## 背景

当前仓库的稳定路径是一套单用户 Windows Service wrapper。它解决了很多真实问题：

- OpenClaw 能在后台运行，不必始终挂着前台 PowerShell 窗口
- 包装层已经提供安装、启动、停止、重启、状态、诊断和托盘控制
- 通过围绕当前登录用户收紧服务身份，已经避免了一部分 `LocalSystem`、路径漂移和托盘错位问题

但这套设计在继续演进时暴露出越来越明显的宿主错位：

- 产品真实诉求更接近“当前登录用户上下文中的无窗口后台宿主”，而不是“机器级服务优先”
- 为了让 Service 尽量贴近当前用户语义，已经引入了较多补偿逻辑，例如托盘 companion、UAC 控制桥、计划任务重启桥和服务身份约束
- OpenClaw 是上游，我们不能依赖其未来的重启、更新和进程行为保持不变
- 现有 Service 模型还要求当前用户为服务安装设置密码，这与“用户级后台工具”的目标发生冲突

在这些事实下，继续把 Windows Service 作为默认主路径的收益正在下降，而维护复杂度在持续上升。

## 决策

V2 默认宿主不再是 Windows Service，而是**当前用户级后台 Agent**。

V2 第一阶段固定采用以下默认决策：

- 技术主线：`.NET 8 + C#`
- 控制面：`CLI + Tray`
- 托盘技术：`WinForms`
- 宿主模型：无窗口、单实例、当前用户级后台 Agent
- IPC：`Named Pipe`
- 更新策略：`Observe Only`
- 注销策略：`Stop On Sign-out`
- 用户约束：必须支持“当前 Windows 用户没有密码”的场景

V2 的设计目标不是替代 OpenClaw 本体，而是把包装层重新定义为：

- 当前用户级宿主
- 生命周期控制入口
- 状态、日志、健康检查与诊断层
- 旧 Service 模型的迁移承载层

## 为什么原方案在当时是合理的

最初采用 Windows Service 不是错误决策，而是一个非常自然的起点：

- “后台运行”在 Windows 上最容易被联想到 Service
- Service 看上去更正式、更稳定，也更接近“开机长期运行”的直觉
- 在需求尚未完全清晰时，WinSW + PowerShell 是一个能快速落地并提供稳定操作入口的方案

更重要的是，这条路径帮助我们摸清了真实需求和边界：

- OpenClaw 需要后台无窗口运行
- OpenClaw 需要尽量继承当前登录用户的权限和环境
- 多数场景其实不是机器级多用户服务，而是单用户 Windows 桌面运行
- Service 方案的主要复杂度不在功能本身，而在宿主与用户语义之间的冲突

## 为什么现在不再适合作为默认主路径

随着需求逐渐清晰，Windows Service 作为默认宿主出现了以下结构性问题：

- 当前用户语义和 Service 语义天然不一致
- Tray、CLI、Service 生命周期被迫拆成多层补偿逻辑
- 用户注销、提权、计划任务桥接、服务账户和 remembered config 这些问题，都不是 OpenClaw 本身的业务问题，而是宿主模型带来的运维成本
- “需要后台运行”并不等于“必须做成 Service”
- “需要当前用户权限”与“依赖服务密码安装”之间存在直接张力

因此，继续在 Service 主路径上累积复杂度，已经不能代表最优长期方向。

## 被放弃的替代方案

### 继续以 Windows Service 为默认路径

放弃原因：

- 与当前用户级需求不匹配
- 会继续放大密码、提权、托盘桥接和更新协同问题
- 会继续把设计重心锁在系统级宿主而不是用户级宿主

### Node.js / Electron / Tauri 主线

放弃原因：

- 当前主要问题不是页面表达，而是 Windows 本地宿主和生命周期控制
- 浏览器壳不适合作为第一阶段的重点
- Electron/Tauri 等方案在这个项目上会把简单问题重新包装成 UI 技术选型问题
- 相比之下，Windows 托盘、自启、单实例和进程控制更适合优先使用 .NET 原生能力

### Rust 主线

放弃原因：

- Rust 不是不可以，而是与当前问题的主要重心不完全匹配
- 当前目标是 Windows 用户级宿主、托盘和生命周期稳定性，而不是性能或底层系统能力极限
- 与 .NET 相比，第一阶段的开发效率和 Windows 桌面宿主自然度都不占优

## 结果

V2 之后，仓库将以“当前稳定 Service 方案 + 下一阶段用户级 Agent 设计”并存的方式推进：

- 旧的 Windows Service 方案短期继续保留，作为 fallback
- V2 设计文档先冻结决策，再进入实施
- 后续代码实施默认围绕 `.NET` 用户级后台 Agent 展开

## 影响

正向影响：

- 架构重心与真实需求对齐
- 不再要求为后台运行强行使用 Service 安装模型
- 无密码当前用户支持成为架构约束，而不是补丁式兼容
- Tray 和 CLI 可以回归控制面角色，不再充当宿主补偿层

需要接受的成本：

- 需要新增一条完整的用户级宿主实现路径
- 需要在过渡期维护旧 Service 方案和新设计文档
- 需要重新定义状态模型、IPC 和迁移策略

## 后续

这个 ADR 生效后，后续文档与实施应遵循以下顺序：

1. 完成 V2 架构蓝图
2. 完成 V2 迁移计划
3. 基于文档冻结后的决策拆分实施里程碑
4. 在不破坏旧 Service fallback 的前提下推进用户级 Agent 实现
